# 1. Введение
**Социальные сети** — важный элемент нашей повседневной жизни. Они приносят человеку много пользы, однако, одними из самых полезных возможностей можно считать следующие:
  * Поиск помощников для учебного проекта
  * Связь со старшими курсами для получения консультаций по различным вопросам
  * Привлечение внимание к учебным и развлекательным мероприятиям
  К сожалению, основная масса социальных сетей не приспособлены к этому, из-за чего приходится решать данную проблему способами, которые не всегда бывают эффективны и / или удобны. 

## 1.1 Глоссарий
- **Модерировать** — редактировать, убирать из общей выборки сущностей в соответсветвии с определёнными правилами.

- **Интересы** — конечное множество тэгов в анкете Пользователя, по которому осуществляется подбор собеседников, например, “ASP.NET”, “Дискретная математика”. Пользователь вправе выбрать количество тэгов $>= N$.

- **Совпадение интересов** — специальное событие, которое происходит в случае, если у два пользователя взаимно положительно оценили профили друг друга.

## 1.2 Описание предметной области

Текущая работа посвящена решению проблемы студентов, преподавателей и сотрудников ДВФУ — нехватки возможности быстро находить собеседника (помощника, наставника). Существуют следующие решения данной проблемы:
  * Применение уже существующих Мессенджеров и/или Социальных сетей
  * Применение сервисов знакомств
  * Применение досок объявлений

Очевидно, что предложенные методы являются либо устаревшими, либо недостаточно сильно ориентированы на решение конкретной задачи.
Кроме того, можно утверждать, что текущее решение этой проблемы может применяться не только в ДВФУ, но и в других крупных ВУЗах.

## 1.3 Неформальная постановка задачи
Требуется реализовать проект социального сервиса для студентов, работников и преподавателей ДВФУ, имеющий следующие возможности:

  * Поиск помощников для создания проекта, путем создания соответствующего объявления.
  * Связь со старшими курсами своего или других образовательных программ с целью получения консультации по учебным или организационным моментам или другой полезной информации.
  * Привлечение внимания к участию в учебных и развлекательных мероприятиях путем создания соответствующего объявления. 


## 1.4 Обзор существующих методов решения
На данный момент, для решения проблем, описанных п. 2.2, используются глобальные социальные сети, в которых создаются соответствующие чаты и / или сообщества для конкретных социальных групп. Однако, такой подход довольно неэффективен, так как многие могут не узнать или потерять связь с теми или иными сообществами / чатами, и соответственно, не иметь возможности решить описанные в п. 2.3 проблемы. Настоящий проект может решить данную проблему, так как вся необходимая информация будет публиковаться в одном приложении и будет видна всем желающим людям.

# 2. Функциональные требования

*ИМЕТЬ ДВУСМЫСЛЕННО ЗВУЧИТ*

**Разработанная система должна иметь следующие категории пользователей:**
- Администратор
- Пользователь

*РАЗДЕЛИТЬ ОПИСАНИЕ НА МАРКИРОВАННЫЕ СПИСКИ*

Система должна позволять **Администратору** модерировать (удалять, убирать из общей выборки) публикации, профили Пользователе через админ-панель (раздел в приложении доступный соотвественно только Администраторам), скрывать себя из общей выборки Пользователей, а также совершать все действия, доступные Пользователю. 

Система должна позволять **Пользователю** создавать личный профиль (регистрироваться, а также входить), заполнять его произвольными данными; небходимыми для ввода данными являются логин и пароль; имя и фамилия, возраст, курс, институт (школа), направление подготовки, Интересы, электронная почта (ДВФУ), аватар, описание являются необязательными для ввода; система должна позволять пользователю изменять данные профиля через опрелённый раздел, осуществлять поиск других Пользователей по Интересам, писать сообщения другим Пользователям, создавать объявления, видеть объявления других людей.

**Поиск других Пользователей по Интересам** проходит следующим образом: осуществляется запрос к базе данных по специальным критериям — Интересам; производится сортировка полученных данных, причём таким образом, чтобы Пользователи имеющие большее совпадение по количеству общих Интересов имели приоритет над иными (были выше в списке). В каждый момент времени Пользователь видит только один профиль из полученных. Система должна позволять Пользователю отметить профиль как понравившийся или же же пропустить его; а также раскрыть профиль, чтобы увидеть больше данных. В случае, если система не смогла подобрать подходящие анкеты Пользователю выводится сообщение "Не удаётся подобрать профиль с учётом Ваших Интересов". Иначе выводятся непосредственно профили Пользователей.

**Чат с другими Пользователями** происходит следующим образом: система позволяет Пользователю открыть непосредственно Диалог с другим Пользователем, а также ввести в соответвующее поле своё сообщение и отправить его; система позволяет Пользователю видеть сообщения, которые отправил собеседник.

**Создание публикаций** происходит следующим образом: система позволяет Пользователю перейти в необходимый раздел, показывает форму с полем ввода; система позволяет Пользователю ввести в это поле своё сообщение и опубликовать его. После проверки Администратором оно появится в выборке у всех пользователей.

**Просмотр объявлений** происходит следующим образом: система позволяет Пользователю перейти в необходимый раздел и выбрать институт (школу), в рамках которой он желает видеть объявления (по умолчанию выбирается школа из профиля); система позволяет Пользователю видеть все объявления, оставленные другими Пользователями, оценивать их (понижать/повышать рейтинг — числовую оценку любого объявления); система позволяет Пользователю отсортировать объявления по времени публикации/рейтингу.

# 3. Требования к интерфейсу

*ОПУСТИТЬ НЕКОТОРЫЕ ПОДРОБНОСТИ*
*АССИМЕТРИЧНЫЙ ИНТЕРФЕЙС В МАКЕТЕ **FIGMA***
*СДЕЛАТЬ СКРИНШОТ И ОПИСАТЬ В НЁМ КАЖДЫЙ ЭЛЕМЕНТ*

**Интерфейс** приложения представляет собой **Web-интерфейс**. В данном случае будет рассматриваться только клиентская часть, именно она осуществляет взаимодействие между Пользователем и сервером. 

Все страницы приложения построены следующим образом:
- **Навигационная панель**. Всегда прикреплена к левой части страницы. Через неё осуществляется навигация по сайту, т.е., в ней содержатся все страницы, описываемые далее (кроме страницы регистрации). В левой части также содержится логотип сервиса и его название. В правой же содержится имя текущего Пользователя, если он авторизован.
- **Основное содержимое (тело)**. Определяется предназначением непосредственно страницы.
- **Футер**. Находится в самом низу страницы. На нём содержится основная информация об авторах и прочие логотипы.

Определяются следующие страницы приложения:
- **Главная страница (Пользователь не авторизован)**. Через неё осуществляется вход в аккаунт Пользователя; если у Пользователя нет аккаунта, ему предлагается зарегистрировать его. Здесь же содержится описание сервиса и его логотип. Форма для входа в аккаунт содержит поля логина (электронной почты) и пароля, а также кнопку, осуществляющую проверку и отправку данных на сервер. Форма должна осуществлять проверку вводимых данных и выдавать сообщение об ошибках, если таковые имеются. Все поля должны быть заполнены соответствующим образом.
- **Главная страница (Пользователь авторизован)**. Выводится следующая информация о подобранном Пользователе: имя, возраст, Интересы. При нажатии на кнопку "Подробнее", расположенную снизу, показывается год поступления/окончания ВУЗа, уровень образования, образовательная программа, институт (школа), статус. Предусмотрена кнопка "Следующий", расположенная справа, для перехода к следующему профилю.
- **Страница регистрации**. Через неё осуществляется регистрация нового Пользователя. Регистрация происходит в несколько этапов. На первом этапе (странице) предлагается ввести придуманный логин (латиница, без специальных символов) и пароль с его повторением (латиница, без специальных символов). На втором этапе (странице) Пользователю предлагается ввести имя, фамилию, возраст, статус, уровень образования, образовательную программу, институт (школу), год поступления и окончания ВУЗа. Этот этап и последующие можно пропустить (кнопка "Пропустить"). На третьем этапе (странице) содержится набор Интересов, каждый из которых представляет собой интерактивный элемент, который можно выбрать. После его выбора он помечается как выбранный. Все перечисленные данные вводятся также в специальные поля; на каждой из страниц содержится кнопка, осуществляющая отправку данных на сервер.
- **Диалоги**. Эта страница содержит Диалоги с Пользователями (в левой части страницы), которые обозначаются как имена этих Пользователей, и непосредственно содержимое Диалога (в правой части страницы). Тело Диалога представляет собой список сообщений, а также сверху отображено имя собеседника. Сообщения собеседника и Пользователя имеют разные цвета, поскольку необходимо различать.
- **Личные данные**. Эта страница содержит информацию о Пользователе, в том числе, его аватар, также содержится и кнопка, осуществляющая смену аватара Пользователем. Кроме того, содержатся поля с текстом для ввода/изменения информации о Пользователе аналогичные тем, что на странице регистрации.
- **Публикации**. Этот раздел сайта содержит все Публикации. Выбор текущей страницы осуществляется через кнопки с соответствующими номерами; содержатся кнопки для перехода на следующую/предыдущую страницу. Допускается отображение не более одной публикации в ряд. Публикация содержит Школу, от имени которой она отправлена, текст, числовое значение рейтинга, а также кнопки, позволяющие поднять/опустить рейтинг у публикации. Предусмотрена кнопка "Отозваться", после нажатия на которую автор Публикации получает сообщение о том, что текущий Пользователь отозвался на объявление. При нажатии на кнопку "Подробнее", расположенную снизу, Публикация показывается полностью.

## 4.3 Описание проткола
**Протокол HTTPS** (HyperText Transfer Protocol Secure) —  расширение протокола HTTP для поддержки шифрования в целях повышения безопасности. Данные в протоколе HTTPS передаются поверх криптографических протоколов TLS или устаревшего в 2015 году SSL. В отличие от HTTP с TCP-портом 80, для HTTPS по умолчанию используется TCP-порт 443.

#### Идентификация сервера
**HTTP/TLS** запросы генерируются путём разыменования URI, вследствие чего имя хоста становится известно клиенту. В начале общения, сервер посылает клиенту свой сертификат, чтобы клиент идентифицировал его. Это позволяет предотвратить атаку посредника. В сертификате указывается URI сервера. Согласование имени хоста и данных, указанных в сертификате, происходит в соответствии с протоколом RFC 2459.

#### Идентификация клиента
Обычно сервер не располагает информацией о клиенте, достаточной для его идентификации. Однако для обеспечения повышенной защищенности соединения используется так называемая two-way authentication. При этом сервер после подтверждения его сертификата клиентом также запрашивает сертификат. Таким образом, схема подтверждения клиента аналогична идентификации сервера.

### Уязвимости HTTPS
#### Совместное использование HTTP и HTTPS
Когда сайты используют смешанную функциональность **HTTP** и **HTTPS**, это потенциально приводит к информационной угрозе для пользователя. Например, если основные страницы некоторого сайта загружаются, используя HTTPS, а CSS и JavaScript загружаются по HTTP, то злоумышленник в момент загрузки последних может подгрузить свой код и получить данные HTML-страницы. Многие сайты, несмотря на такие уязвимости, загружают контент через сторонние сервисы, которые не поддерживают HTTPS. Механизм HSTS позволяет предотвратить подобные уязвимости, заставляя принудительно использовать HTTPS соединение даже там, где по умолчанию используется HTTP.

#### Атаки с использованием анализа трафика
В **HTTPS** были обнаружены уязвимости, связанные с анализом трафика. Атака с анализом трафика — это тип атаки, при которой выводятся свойства защищенных данных канала путем измерения размера трафика и времени передачи сообщений в нём. Анализ трафика возможен, поскольку шифрование SSL/TLS изменяет содержимое трафика, но оказывает минимальное влияние на размер и время прохождения трафика.
